Resumen del flujo

  1. prplMesh simula una red EasyMesh y genera una topología: gateway GW_MASTER, radio wlan0 y radio wlan2.
  2. El adaptador entra en el contenedor de prplMesh y ejecuta beerocks_cli -c bml_conn_map.
  3. El adaptador transforma esa salida en un fichero data_model_easymesh.csv.
  4. tr069-sim arranca como CPE virtual y carga ese CSV como modelo de datos TR-069.
  5. GenieACS recibe el CPE, consulta parámetros y acaba mostrando valores EasyMesh dentro de X_SARA_EasyMesh.

Qué hace el archivo principal

  • mongo crea la base de datos que usa GenieACS.
  • redis crea el servicio auxiliar de GenieACS.
  • genieacs crea el ACS real del laboratorio.
  • easymesh-simulator crea el contenedor donde corre prplMesh.

Si tu archivo principal todavía tiene una línea "8080:3000" dentro de genieacs.ports, puede provocar conflicto con otra web. La solución limpia es dejar GenieACS en 3000:3000 y servir esta documentación en 8181:80.
docker-compose.override.yml - ampliación del laboratorio
services:
  easymesh-tr069-adapter:
    build:
      context: ./adapter
      dockerfile: Dockerfile
    container_name: easymesh-tr069-adapter
    depends_on:
      - easymesh-simulator
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - ./tr069-data:/data
    restart: "no"

  tr069-sim:
    build:
      context: ./tr069-sim
      dockerfile: Dockerfile
    container_name: tr069-sim
    depends_on:
      - genieacs
      - easymesh-tr069-adapter
    volumes:
      - ./tr069-data:/data:ro
    restart: unless-stopped

  pag-web:
    image: nginx:alpine
    container_name: pag-web
    ports:
      - "8181:80"
    volumes:
      - ./pag-web/index.html:/usr/share/nginx/html/index.html:ro
    restart: unless-stopped

Qué añade el override

  • easymesh-tr069-adapter: lee prplMesh y genera el CSV TR-069.
  • /var/run/docker.sock: permite ejecutar docker exec contra easymesh-simulator.
  • ./tr069-data:/data: carpeta compartida para la salida BML y el modelo CSV.
  • tr069-sim: CPE virtual que carga el CSV y se registra en GenieACS.
  • pag-web: sirve esta documentación con Nginx en http://localhost:8181.

03_genieacs_acs

GenieACS es el servidor ACS. En este laboratorio recibe al CPE virtual y guarda los parámetros EasyMesh.

ACS / CWMP / NBI
Puerto 3000Interfaz web de GenieACS. Aquí se ve el dispositivo registrado.
Puerto 7547Endpoint CWMP/TR-069. El CPE virtual se conecta aquí.
Puerto 7557API NBI. Se usa para consultar dispositivos y crear tareas.
Comprobación de GenieACS 
curl.exe -i http://localhost:3000
curl.exe -i http://localhost:7547/
curl.exe -i http://localhost:7557/devices

Qué respuesta es normal

  • http://localhost:3000: debe abrir la interfaz web.
  • http://localhost:7547/: puede devolver 405 Method Not Allowed. Eso es correcto porque CWMP espera peticiones POST.
  • http://localhost:7557/devices: devuelve un JSON con los dispositivos registrados.
El dispositivo final esperado en GenieACS es 535241-EasyMeshVirtualCPE-200001. Ese identificador sale de Manufacturer OUI 535241, ProductClass EasyMeshVirtualCPE y SerialNumber 200001.

04_prplmesh

prplMesh/EasyMesh es la fuente de datos. En vez de inventar los valores del CPE, el laboratorio los lee desde la topología simulada.

EasyMesh / BML
Comando para consultar la topología BML
docker exec -it easymesh-simulator bash -lc "export LD_LIBRARY_PATH=/opt/prplmesh/build/install/lib:/opt/prplmesh/build/out/lib:/usr/local/lib:/usr/lib; cd /opt/prplmesh/build/install/bin; ./beerocks_cli -c bml_conn_map"
Salida BML esperada 
bml_connect: return value is: BML_RET_OK, Success status
bml_nw_map_query: return value is: BML_RET_OK, Success status

GW_BRIDGE: name: GW_MASTER, mac: 06:8e:01:be:ef:29, ipv4: 0.0.0.0
    ETHERNET: mac: 06:8e:01:be:ef:2a
    RADIO: wlan0 mac: 2e:ac:27:ab:5d:92, ch: 1, bw: 40L, freq: 2412MHz
    RADIO: wlan2 mac: 0e:92:da:b8:03:bf, ch: 149, bw: 80H, freq: 5745MHz
bml_disconnect: return value is: BML_RET_OK, Success status

Qué significan esos valores

  • GW_MASTER: nombre del gateway EasyMesh detectado.
  • 06:8e:01:be:ef:29: MAC del gateway.
  • wlan0: primera radio, canal 1, frecuencia 2412 MHz, banda 2.4 GHz.
  • wlan2: segunda radio, canal 149, frecuencia 5745 MHz, banda 5 GHz.
  • BML_RET_OK: confirma que la consulta a prplMesh funciona.

05_adaptador

El adaptador es la pieza que une EasyMesh con TR-069. Lee prplMesh, genera el CSV y permite que GenieACS vea datos EasyMesh.

EasyMesh → TR-069
adapter/Dockerfile
FROM docker:27-cli

RUN apk add --no-cache bash grep sed coreutils

COPY generate-easymesh-datamodel.sh /usr/local/bin/generate-easymesh-datamodel.sh

RUN sed -i 's/\r$//' /usr/local/bin/generate-easymesh-datamodel.sh && \
    chmod +x /usr/local/bin/generate-easymesh-datamodel.sh

CMD ["/usr/local/bin/generate-easymesh-datamodel.sh"]

Explicación línea a línea del Dockerfile

  • FROM docker:27-cli: usa una imagen que trae el cliente Docker. Esto permite ejecutar docker exec desde el adaptador.
  • apk add: instala herramientas para ejecutar y parsear texto.
  • COPY: mete dentro de la imagen el script generador.
  • sed -i 's/\r$//': elimina saltos de línea de Windows.
  • chmod +x: hace ejecutable el script.
  • CMD: ejecuta el generador al arrancar el contenedor.
generate-easymesh-datamodel.sh - lógica resumida
#!/usr/bin/env bash
set -e

OUT_DIR="/data"
BML_OUT="$OUT_DIR/bml_conn_map_para_tr069.txt"
CSV_OUT="$OUT_DIR/data_model_easymesh.csv"

docker exec easymesh-simulator bash -lc "export LD_LIBRARY_PATH=/opt/prplmesh/build/install/lib:/opt/prplmesh/build/out/lib:/usr/local/lib:/usr/lib; cd /opt/prplmesh/build/install/bin; ./beerocks_cli -c bml_conn_map" > "$BML_OUT"

cat > "$CSV_OUT" < TR-069 finalizado correctamente."
exit 0

Qué hace realmente

  1. Ejecuta beerocks_cli -c bml_conn_map dentro de easymesh-simulator.
  2. Guarda la salida exacta en /data/bml_conn_map_para_tr069.txt.
  3. Extrae gateway, MAC, radios, canales, anchos de banda y frecuencias.
  4. Crea /data/data_model_easymesh.csv.
  5. Ese CSV será leído por tr069-sim.

06_tr069_sim

El CPE virtual es el dispositivo que GenieACS gestiona. No es un router físico, pero se comporta como un CPE TR-069 para el ACS.

CPE virtual
tr069-sim/Dockerfile 
FROM node:14-bullseye

RUN apt-get update && apt-get install -y \
    git \
    ca-certificates \
    curl \
    && rm -rf /var/lib/apt/lists/*

RUN git clone https://github.com/genieacs/genieacs-sim.git /opt/genieacs-sim

WORKDIR /opt/genieacs-sim

RUN npm install

RUN chmod +x ./genieacs-sim

CMD ["bash", "-lc", "until test -f /data/data_model_easymesh.csv; do echo 'Esperando modelo EasyMesh generado por el adaptador...'; sleep 5; done; until curl -s -o /dev/null -w '%{http_code}' http://genieacs:7547/ | grep -q '405'; do echo 'Esperando a GenieACS CWMP en genieacs:7547...'; sleep 5; done; echo 'GenieACS CWMP disponible. Arrancando CPE virtual EasyMesh...'; ./genieacs-sim --acs-url http://genieacs:7547/ --data-model /data/data_model_easymesh.csv --processes 1 --serial 200001"]

Qué hace este contenedor

  • Instala Node.js y clona genieacs-sim.
  • Espera a que exista /data/data_model_easymesh.csv.
  • Espera a que GenieACS responda en http://genieacs:7547/.
  • Arranca el simulador con --serial 200001.
  • Usa --data-model para exponer los parámetros generados.
En los logs debe aparecer algo como: Simulator 200001 started y GenieACS CWMP disponible. Arrancando CPE virtual EasyMesh....

Qué representa la animación

  1. prplMesh genera la topología EasyMesh.
  2. Adaptador lee BML y crea data_model_easymesh.csv.
  3. tr069-sim carga el CSV y actúa como CPE virtual.
  4. GenieACS recibe la sesión CWMP/TR-069.
  5. MongoDB guarda los parámetros y estado.
  6. Redis apoya el funcionamiento interno.
  7. tcpdump captura tráfico para evidencias.
